`localStorage` is a key/value store that only speaks strings. Every time you save a `Date`, a `Set`, or a `Map`, you have to remember to serialize it on the way in and reconstruct it on the way out. There is no built-in way to expire stale data, and there is nothing to stop you from reading back a value that no longer matches the shape your code expects.

greatstorage is a thin wrapper around `localStorage` (or any `Storage`-compatible backend) that solves these problems without replacing the familiar API.

## How it works

Every value you store is wrapped in an **envelope** before it reaches `localStorage`:

```json
{
  "__gs": true,
  "version": 1,
  "value": "<serialized value>",
  "expiry": 1741996800000
}
```

- `__gs` and `version` let greatstorage identify its own entries and ignore unrelated keys in the same storage backend.
- `value` holds the output of the serializer. By default greatstorage uses [devalue](https://github.com/Rich-Harris/devalue), which preserves `Set`, `Map`, `Date`, `RegExp`, `BigInt`, `undefined`, and circular references that `JSON.stringify` would either mangle or throw on.
- `expiry` is an optional Unix timestamp in milliseconds. When a read finds an expired envelope it removes the entry and returns `null`.

On read, greatstorage unwraps the envelope, checks the expiry, deserializes the value, and optionally validates it against a schema before returning it to you.

<Note>
  greatstorage only touches entries it created. Keys written directly to `localStorage` by other code are not affected by `clear()` or `clearExpired()`.
</Note>

## Package exports

greatstorage ships two entry points:

| Import | Description |
|---|---|
| `greatstorage` | Main entry point with devalue bundled. Use this for browser apps. |
| `greatstorage/core` | Core functionality without devalue. You must provide a `serializer` option. Use this to keep bundle size minimal or to supply a custom serializer. |
